# 项目重构计划

## 概述
本文档详细记录了 Legends Never Die 项目的重构计划，旨在将项目从个人练手项目提升至商用级标准。

---

## 阶段一：工程化基础（优先级：最高）
**目标**：建立规范的工程化流程，确保项目可维护、可复现。

### 任务清单
- [ ] 1.1 完善 .gitignore
  - 添加 \dist/\、\in/\、\elease/\ 目录
  - 添加 IDE 配置文件（.idea/、.vscode/、*.iml）
  - 添加操作系统临时文件（Thumbs.db、.DS_Store）
  
- [ ] 1.2 统一静态资源
  - 将所有静态资源统一到根目录 \ssets/\ 下
  - 删除 \static/\、\or_java/dist/static/\、\or_java/release/static/\ 重复目录
  - 更新 Java 和 Web 端的资源加载路径
  
- [ ] 1.3 配置依赖管理
  - Web 端：运行 \
pm audit\ 并修复安全漏洞
  - Java 端：配置 Maven 依赖版本管理
  - 添加依赖锁定文件（package-lock.json 保留）
  
- [ ] 1.4 建立分支策略
  - 创建 \develop\ 分支作为开发分支
  - 创建 \eature/*\ 分支用于功能开发
  - 创建 \hotfix/*\ 分支用于紧急修复
  - 更新 README 说明分支使用规范

### 验收标准
- [ ] .gitignore 规则完整，无构建产物被提交
- [ ] 静态资源仅有一份，位于 \ssets/\ 目录
- [ ] 依赖扫描无高危漏洞
- [ ] 分支结构清晰，README 包含分支规范

---

## 阶段二：代码组织重构（优先级：高）
**目标**：建立清晰的代码架构，降低耦合度，提升可维护性。

### 任务清单
- [ ] 2.1 Java 端包结构重构
  - 创建包结构：
    - \com.legends.game.core\ - 核心游戏逻辑
    - \com.legends.game.entity\ - 游戏实体（Hero、Enemy、Bullet 等）
    - \com.legends.game.input\ - 输入处理
    - \com.legends.game.render\ - 渲染相关
    - \com.legends.game.audio\ - 音频管理
    - \com.legends.game.config\ - 配置和常量
    - \com.legends.game.util\ - 工具类
  - 将 \Start.java\ 拆分为多个类
  - 将所有硬编码数值提取到配置类
  
- [ ] 2.2 Web 端模块重构
  - 创建目录结构：
    - \js/core/\ - 核心游戏逻辑
    - \js/entities/\ - 游戏实体
    - \js/input/\ - 输入处理
    - \js/render/\ - 渲染相关
    - \js/audio/\ - 音频管理
    - \js/config/\ - 配置和常量
    - \js/utils/\ - 工具类
  - 将 \game.js\ 拆分为多个模块
  - 将所有硬编码数值提取到配置文件
  - 考虑引入 TypeScript（可选但推荐）

### 验收标准
- [ ] Java 端有清晰的包结构，无默认包类
- [ ] \Start.java\ 拆分为至少 10 个职责单一的类
- [ ] Web 端有清晰的目录结构
- [ ] \game.js\ 拆分为至少 8 个职责单一的模块
- [ ] 所有游戏数值统一在配置文件中管理

---

## 阶段三：核心逻辑统一（优先级：高）
**目标**：消除双端实现割裂，确保玩法一致。

### 任务清单
- [ ] 3.1 设计数据驱动配置
  - 创建统一的配置格式（JSON/YAML）
  - 将升级卡池、数值公式、生成规则等配置化
  - 实现配置加载器（Java 和 Web 端各一个）
  
- [ ] 3.2 提取共享逻辑（二选一）
  - **方案 A**：选择 Web 端为主实现，Java 端仅作为历史保留
  - **方案 B**：选择 Java 端为主实现，通过编译到 WebAssembly 或 JS 实现 Web 端
  - **推荐方案 A**，维护成本更低
  
- [ ] 3.3 建立一致性测试
  - 编写核心玩法单元测试（覆盖升级、碰撞、生成等）
  - 确保双端（如保留）通过相同测试用例

### 验收标准
- [ ] 所有游戏规则通过配置文件定义，代码中无硬编码规则
- [ ] 核心玩法测试覆盖率 >= 70%
- [ ] 双端（如保留）通过相同测试用例

---

## 阶段四：测试体系建设（优先级：高）
**目标**：建立完整的测试体系，保证代码质量。

### 任务清单
- [ ] 4.1 Web 端测试
  - 配置测试框架（Jest/Vitest）
  - 编写单元测试（覆盖 utils、config、核心逻辑）
  - 编写集成测试（覆盖游戏流程）
  - 配置测试覆盖率报告
  
- [ ] 4.2 Java 端测试
  - 配置 JUnit
  - 编写单元测试
  - 配置测试覆盖率报告（JaCoCo）
  
- [ ] 4.3 E2E 测试
  - 配置 Playwright/Cypress
  - 编写端到端测试（覆盖完整游戏流程）

### 验收标准
- [ ] Web 端单元测试覆盖率 >= 60%
- [ ] Java 端单元测试覆盖率 >= 50%
- [ ] 至少 5 个 E2E 测试用例
- [ ] 测试在 CI 中自动运行

---

## 阶段五：CI/CD 建设（优先级：中）
**目标**：建立自动化构建、测试、部署流程。

### 任务清单
- [ ] 5.1 配置 GitHub Actions
  - 创建构建流程（Java 和 Web）
  - 创建测试流程
  - 创建代码质量检查流程（ESLint、Checkstyle）
  
- [ ] 5.2 配置部署流程
  - Web 端：配置自动部署到 GitHub Pages 或其他托管平台
  - Java 端：配置自动构建可执行文件
  
- [ ] 5.3 配置发布流程
  - 自动生成 CHANGELOG
  - 自动创建 GitHub Release

### 验收标准
- [ ] 每次 PR 自动运行构建和测试
- [ ] 代码质量检查不通过无法合并
- [ ] Web 端自动部署到预览环境
- [ ] 有自动发布流程

## 阶段七：性能与安全优化（优先级：低）
**目标**：优化性能，提升安全性。

### 任务清单
- [ ] 7.1 性能优化
  - 压缩静态资源（图片、音频）

  - 定期更新依赖
  - 配置 Content Security Policy（Web 端）
  - 检查第三方资源版权

### 验收标准
- [ ] Web 端首屏加载时间 < 2s
- [ ] 游戏帧率稳定在 60 FPS
- [ ] 依赖无高危漏洞
- [ ] 所有第三方资源有明确的版权说明

## 风险与应对

| 风险 | 影响 | 概率 | 应对措施 |
|------|------|------|----------|
| 重构过程中引入新 Bug | 高 | 中 | 建立完善的测试体系，每步重构后运行测试 |
| 双端保留导致维护成本高 | 高 | 高 | 尽早决策保留哪一端，建议保留 Web 端 |
| 团队成员不熟悉新架构 | 中 | 中 | 编写架构文档，开展代码 Review |
| 进度延期 | 中 | 中 | 优先级排序，先完成高优先级任务 |

## 备注

- 本计划为迭代式重构，每个阶段完成后可发布一个版本
- 建议先在 \develop\ 分支完成重构，再合并到 \master\
- 重构过程中保持游戏功能可用，避免长时间无法玩
- 定期同步进度，根据实际情况调整计划
